## 2. Синопсис и лор

**Рабочее название:** «Морская Глушь»

**Жанр:** Экзистенциальная притча от первого лица с лёгкими элементами боёвки. Бытовой сюрреализм. Атмосфера «маленького человека».

**Референсы:** Disco Elysium (нарратив), Pathologic 2 (атмосфера, неудобная боёвка), Firewatch (перспектива, одиночество), Kentucky Route Zero (магический реализм), Silent Hill 2 (отложенное самопознание).

**Сеттинг:**
- Приморский край, Дальний Восток, побережье Японского моря
- Тайга, туманы, ржавые остовы кораблей, крики чаек
- «Пятёрочка» на въезде — единственный светящийся центр цивилизации
- Начало 2000-х. Без элементов будущего.

**Природа аномалий:**
- Не Лавкрафт. Бытовой сюрреализм, системный сбой реальности.
- Место истончилось от заброшенности.
- Разные аномалии: не агрессивные и уютные (искажения времени, чайки с человеческими голосами) + агрессивные (твари из тумана).
- Дружелюбные монстры хранят секреты — их можно убить, но морально дорого.

- Местные жители, рыбаки, безумцы, возможно культисты

## 4. Четыре финала

| Финал | Условие | Тон |
|---|---|---|
| **1. Отрицание / Боевой** | Житейская < −30, Азарт не пробит | Тяжёлый, но морально сильный |
| **2. По течению** | −30 ≤ Житейская ≤ +30, Азарт не пробит | Меланхолично-нейтральный |
| **3. Смирение / Растворение** | Житейская > +30, Азарт не пробит | Уютное растворение |
| **4. Стервятник / Азарт** | Азарт пробил скрытый порог | Тревожно двойственный |

**Дизайнерское намерение по 4-му финалу:** игрок должен прийти к нему **неосознанно**, через скользкий склон мелких компромиссов.

## 5. Система Личности

> Полная техническая спецификация — `Personality_Design.md` v2.0.

### 5.1 Три шкалы

Все в диапазоне **[−100, +100]**, старт 0:

| # | Ось | Роль |
|---|---|---|
| **1** | **Житейская** (Эхо войны ↔ Фатализм) | Магистральная. Определяет путь, финал, сложность механических проверок. |
| **2** | **Поведенческая** (Приказ ↔ Своеволие) | Разблокировщик противоречий в диалогах + модификатор тона + **эффективный модификатор для механических проверок** (см. §5.3). |
| **3** | **Поэтичная** (Трезвость ↔ Иллюзия) | Косметика реплик + гейт «глубоких» веток + визуальное оформление механик. |

### 5.2 Правила доступности вариантов в диалогах

- Основа: `|Житейская| ≤ 20` → доступны 3 варианта; 20..60 → нейтральный + вариант в тон; >60 → только вариант в тон.
- Поведенческая при большом уклоне (>40) в противоположную от Житейской сторону → разблокирует «противоречивый» вариант.
- Поэтичная в обычных диалогах — только окраска текста; в «глубоких» — как основная шкала.

### 5.3 Формула эффективной шкалы (для механических проверок)

**Для механических проверок** (стрельба, будущие проверки на ловкость/реакцию/что угодно):

```
эффективная_Житейская = реальная_Житейская + (Поведенческая × 0.2)
```

Затем клэмп к [−100, +100].

**Пример:**
- Житейская реальная: −60 (кап Эхо)
- Поведенческая: +60 (Своеволие)
- Бонус: +60 × 0.2 = +12
- Эффективная Житейская для стрельбы: −60 + 12 = **−48**

Смысл: Своеволие «толкает в сторону Фатализма», Приказ — «толкает в сторону Эха». Игрок с высокой Поведенческой в любую сторону сдвигает свою механическую эффективность.

**Поэтичная не участвует в этой формуле** — она про восприятие, не про действие.

## 6. Шкала Усталости / Азарта (отложено)

Двойное дно:
- Внешне: полоска в углу без подписи, восстанавливается сигаретами/алкоголем/наркотой
- Скрыто: имеет **порог**, пробиваемый «подлыми действиями» (эгоистическая выгода с ущербом другому)
- После пробития — появляется подпись **«Азарт»**, закрываются 3 обычных финала, открывается путь Стервятника

**Реализация — после базовой Системы Личности.**

## 7. Курение (утверждённая модель)

- **Курить можно где угодно** — не по местам
- Активируется по клавише / через меню
- Восстанавливает шкалу Усталости
- **Единственный источник явной рефлексии** — только во время курения генерируется внутренний монолог героя на основе последних событий и шкал
- Иногда всплывает окошко «Сигарета не помешает» — предложение от игры, игрок может согласиться/отказаться
- **Маркер-места** (крыльцо, скамейка у моря) — бонусный слой: особые монологи или скрытые аномалии. Необязательны для прохождения.
- **Курение — точка сохранения** (планируется, обсуждается): выкурил → игра сохранилась

## 8. Пистолет как нарративная функция

### 8.1 Философия

Пистолет — **не оружие в шутерном смысле**. Это **моральный груз** и **функция вызова сцены**.

- **Игрок не может достать пистолет по своей воле** и побегать с ним. Пистолет невидим, всегда в кармане.
- Расчёт: **5-10 столкновений на всю игру**, **5-8 патронов** максимум.
- **Каждый выстрел — это сцена**, а не механика.

### 8.2 Как активируется

В ключевых сценах (конфронтации, встречи с врагами, некоторые аномалии) в диалоговом окне появляется **специальный тип выбора**: «Достать пистолет».

Пример:
```
[Пьяный незнакомец подходит]
* Уйти (Фатализм +3)
* Возразить (Эхо -2)
* [Достать пистолет]  ← открывает мини-игру
```

Только выбор этого варианта запускает мини-игру стрельбы.

### 8.3 Мини-игра стрельбы

Короткая (~2-4 секунды) сценарная последовательность действий. Сложность и оформление зависят от шкал.

**Сложность попадания** — определяется **эффективной Житейской** (§5.3):

| Эффективная Житейская | Сложность | Обоснование |
|---|---|---|
| **−100..−60** (глубокое Эхо) | **Легко** | Профессиональная холодность — руки помнят войну, туннель фокуса |
| **−60..−30** (переход, флешбеки) | Сложно (переход) | «Возвращение на фронт» — герою дурно, руки не слушаются |
| **−30..0** | **Пик сложности** | Обычный человек, интернализующий позицию, но не удержавший её |
| **0..+30** | **Пик сложности** | То же с другой стороны — сопротивление, дрожь |
| **+30..+60** (переход) | Сложно (переход) | Начало отпускания, «зачем всё это» |
| **+60..+100** (Смирение) | **Легко** | Обыденность — стрельба как ветер, как дождь. Уже не сопротивляешься. |

Кривая: **U-образная с двумя пиками на ±30**, дно легкости в центре только теоретически (быстро проскакивается на старте, пики стоят близко), реальная лёгкость только на краях (>±60).

**Философская модель:** легко убивать может либо **сломанный** (Эхо), либо **отпустивший** (Смирение). Обычный человек — не может.

### 8.4 Визуал стрельбы — по Поэтичной

Независимо от сложности:
- **Иллюзия > +40** — метафоричное искажение, красивые цвета, цветы вместо крови, замедленное падение, поэтические образы
- **Трезвость < −40** — холодный реализм, всё чётко, кровь как кровь, никаких прикрас
- **Около нуля** — нейтральный визуал

### 8.5 Тон движений — по Поведенческой

- **Своеволие > +40** — дерганные движения, спешка, азарт
- **Приказ < −40** — размеренная процедура, чёткие шаги, «как учили»
- **Около нуля** — растерянные, неопределённые движения

### 8.6 Звук

Не искусственные подчёркивания, а **фильтры на звуки окружения**:
- **Сильное Эхо (< −60):** сильное приглушение — «туннель профессионала»
- **Умеренное Эхо (−60..0):** нарастание тревожных обертонов в обычных звуках
- **Умеренный Фатализм (0..+60):** тревожные звуки на пике, потом спадают
- **Смирение (> +60):** звуки природы возвращаются, обычный мир, обыденность

### 8.7 Моральный вес выстрелов

| Событие | Житейская | Азарт |
|---|---|---|
| Убийство агрессивного (сам напал) | 0 | 0 |
| Убийство агрессивного (можно было уйти) | −2 | 0 |
| Пощада агрессивного | +3 | 0 |
| Убийство мирного в гневе | −20 | 0 |
| Убийство мирного ради выгоды | −20 | **+25** |
| Убийство дружелюбного монстра за секрет | −15 | +10 |
| Не выстрелил, разошлись словами | +5 | 0 |

## 9. Экономика шкал

Стартовые значения, тюнятся на плейтестах. **Величина сдвига задаётся в месте применения** (в Ink-теге или при вызове API), не через константы.

| Действие | Сдвиг |
|---|---|
| Малый выбор в тон | ±1..±3 |
| Значимый моральный | ±5..±8 |
| Ключевое сюжетное | ±10..±15 |
| Убийство мирного | −20 |
| Убийство дружелюбного монстра | −15 |
| Ритуал (покурил, посмотрел на туман) | +1 в Фатализм |

Клэмп к [−100, +100]. Никаких `SmallShift`/`MediumShift`/`LargeShift` **констант**.

**Никаких переходов между сценами.**

## 12. Архитектура проекта

```
Assets/_Project/
├── Docs/                ← дизайн-документы
├── Materials/           ← материалы (пока грейбокс)
├── Prefabs/             ← префабы (Player как префаб — готов)
├── Scenes/              ← основная сцена
├── Data/                ← ScriptableObject-конфиги (PlayerConfig, InteractionConfig)
├── Scripts/
│   ├── Player/          ← Input, Movement, CameraLook, HeadBob, Footsteps, Config
│   ├── Interaction/     ← Interactable, PlayerInteractor, TestInteractable, Config
│   ├── UI/              ← InteractionPromptUI (+ будущее)
│   ├── Personality/     ← PersonalityAxis, PersonalityData, PersonalityManager
│   ├── Editor/          ← PlayerBuilder
│   ├── Input/           ← .inputactions + сгенерированный C#
│   └── (Dialogue, World, Sanity, Smoking, Combat — будущее)
├── Textures/            ← прототипные клетчатые текстуры
├── Localization/        ← позже
└── Audio/               ← позже
```

**Разделение design-data / player-data:**
- **Design-data** (стартовые значения, скорости, чувствительность) → **ScriptableObject** в `Data/`. Вшивается в билд.
- **Player-data** (текущие значения шкал, позиция, инвентарь) → **обычные C# классы** в runtime + JSON-сохранение в `Application.persistentDataPath`.
- **Никогда не смешивать.**

## 14. Правила разработки

### Общие принципы

1. **KISS** — простое решение всегда предпочтительнее сложного.
2. **YAGNI** — не делать то, что не нужно **прямо сейчас**.
3. **Разделение design-data / player-data** (§12).
4. **Инкапсуляция**: никаких публичных полей.
5. **Одна ответственность на класс.**
6. **Настройки — через ScriptableObject** для design-data. Никаких магических чисел.
7. **Комментарии на русском** — «зачем», не «что».

### Работа с AI

1. **Новый чат для каждой сессии** — контекст должен быть чистым.
2. **Обновление GDD и Personality_Design — только по явному согласию разработчика.**
3. **В новом чате Kilo дать актуальный GDD** как единственный источник истины.
4. **Промпт короткий и конкретный.** Явно указывать, что делать и **чего НЕ делать**.
5. **AI склонен переусложнять.** Явно писать «сделай минимум, без Х».
6. **После каждой сессии — проверка + коммит.**
7. **Ctrl+S в Unity перед каждым коммитом.**

### Нейминг

- `PascalCase` — классы, свойства
- `camelCase` — параметры, локальные переменные
- `_camelCase` — приватные поля
- `UPPER_CASE` — константы
- Namespaces: `Glush.*`

## 16. Реактивность NPC (два уровня)

**Философия:** NPC — не декорации. Они замечают действия игрока и реагируют репликами, даже когда это ничего не меняет в сюжете. Это ключ к живости мира — приём Disco Elysium, Kentucky Route Zero, Undertale.

**Два принципиально разных типа реакций.** Реализуются как **независимые системы**.

### 16.1 Диалоговые реакции (активные)

**Инициатор:** игрок, нажимает E на NPC.

**Формат:** полноценный диалог через `InkDialogueRunner`:
- Фиксация камеры
- Отбор управления
- Полноэкранное диалоговое окно
- Ветвление, выборы, влияние на шкалы

**Контекст:** «входной узел» .ink-файла NPC сам решает, какая ветка играется:
- Первая встреча (никогда не говорил)
- Обычный визит (уже встречались, ничего особого)
- Слишком частые подходы («у тебя всё нормально?»)
- После долгого отсутствия («давно не виделись»)
- После странного поведения в прошлый раз («в прошлый раз ты...»)
- И т.д. — расширяется по мере разработки

**Реализация:** один `.ink` на одного NPC. Внутри — узел `<npc>_entry` с ветвлением по игровым переменным.

**Пример структуры (см. §17.2).**

### 16.2 Пассивные реакции (мир живой)

**Инициатор:** мир, автоматически при действиях игрока рядом с NPC.

**Формат:** короткая всплывающая реплика над NPC («баббл» на 3-5 секунд):
- **НЕ** отбирает управление
- **НЕ** фиксирует камеру
- **НЕ** открывает диалог
- **НЕ** влияет на шкалы (обычно)

**Триггеры (стартовый список, расширяется):**
- Долгий взгляд на NPC (стоишь и смотришь > X секунд)
- Прыжки рядом с NPC
- Кружение вокруг
- Спам взаимодействиями (несколько раз E за короткое время — сработает уже в диалоге)
- Открытие/закрытие двери рядом много раз
- Долгое молчание в присутствии NPC (без диалога)

**Приоритет и повторы:** случайная выборка из подходящих реакций для триггера. Одна и та же реакция не повторяется подряд.

**Реализация:** отдельная система, **не через Ink**:
- Компонент `PlayerObserver` — трекает действия игрока (прыжки, взгляды, ходьбу)
- Компонент `NpcPassiveReactor` на каждом NPC — подписывается на события, знает своё расстояние до игрока, выбирает и показывает баббл
- ScriptableObject `NpcReactionSet` — набор коротких фраз по типам триггеров для конкретного NPC

**Управляемость:**
- Легко **отключается** для конкретного NPC (снял компонент)
- Легко **выключается временно** для всех (в кульминационных сценах, чтобы не отвлекало)
- Не влияет на диалоговую систему

**Пример конфига NPC:**
```
Cashier Reactions:
├── LongStare:  ["Что-то не так с моим лицом?", "Хватит уже.", "..."]
├── JumpsNear:  ["Успокойся.", "Ты в порядке?"]
├── DoorSpam:   ["Двери подводят?"]
```

### 16.3 Разница в трудозатратах

| Аспект | Диалоговая | Пассивная |
|---|---|---|
| Форма | Сценарий (.ink) | Массив коротких фраз |
| Трудозатраты | Высокие (полноценный текст) | Низкие (одна строка) |
| Реализация в проекте | Основная нарративная система | Приятный полироч |
| Обязательна для NPC? | Да (иначе NPC не разговаривает) | Нет (не у всех NPC есть) |

### 16.4 Порядок реализации

1. Диалоговые реакции — базовая механика, обязательная для вертикального среза
2. Пассивные реакции — **позже**, когда основной каркас работает. Отдельная сессия.

## 17. Организация Ink-файлов

**Правило:** один `.ink` на один персонаж (или на одну сцену), внутри — самодокументирующееся ветвление.

### 17.1 Структура папок

```
Assets/_Project/Ink/
├── _debug/                    ← временные тестовые диалоги (не игровой контент)
│   └── dev_test.ink
├── _lib/                      ← общие переменные и макросы (создаётся по мере надобности)
│   ├── globals.ink
│   └── tone_macros.ink
├── npcs/                      ← один .ink на NPC
│   ├── cashier.ink
│   ├── fisherman.ink
│   └── ...
├── smoking/                   ← внутренние монологи при курении
│   ├── smoking_default.ink
│   ├── smoking_pier.ink
│   └── ...
├── events/                    ← особые сцены (флешбеки, скриптовые события)
│   ├── first_anomaly.ink
│   └── ...
└── endings/                   ← финалы
    ├── denial.ink
    ├── flow.ink
    ├── surrender.ink
    └── vulture.ink
```

### 17.2 Правила именования

- Имена файлов — **на английском**, **snake_case**
- Имена самодокументирующие: `first_visit.ink` (не `dialog_003.ink`)
- Внутри Ink — узлы на английском (Ink не любит русские идентификаторы)

### 17.3 «Входной узел» — паттерн для NPC

Стандартная структура `.ink` файла на NPC:

```ink
=== <npc_id>_entry ===
// Умный входной узел — сам решает, какая ветка играется
{
    - not visited_<npc>: -> first_visit
    - recent_approaches >= 3: -> too_often
    - time_since_last > 300: -> long_time_no_see
    - else: -> {~usual_1|usual_2|usual_3}
}

=== first_visit ===
...
-> DONE

=== too_often ===
...
-> DONE

// ... остальные ветки
```

**NPC-триггер запускает только `<npc_id>_entry`** — не выбирает ветки сам. Логика внутри Ink.

### 17.4 Способы вызова диалогов (эволюция)

**Стартовый подход (сейчас):** прямая ссылка в NPC-триггере.
```
NpcDialogueTrigger.Dialogue = cashier.json  // ScriptableObject reference
```

**Когда файлов станет 15-20:** переход на **DialogueRegistry** — один ScriptableObject-таблица «ID → файл». NPC хранит только строковый ID.

**Не делаем сейчас:** «умные триггеры» с массивом диалогов и условиями. Всё ветвление — внутри Ink.

## 19. GameFlags — состояние мира

**Философия:** централизованное хранилище состояния мира — метки событий («знает X», «встречал Y», «убил Z»), счётчики действий, глобальные статусы («время суток», «погода»).

Без этого система невозможна: диалоги должны реагировать на прошлые события, NPC должны помнить встречи, сюжет должен ветвиться.

### 19.1 Что там хранится

**Булевы флаги** (событие было / не было):
- `met_cashier`, `met_fisherman`, ... — встречал ли NPC
- `knows_cashier_is_monster`, `knows_about_anomaly_lighthouse`, ... — узнал ли скрытую правду
- `ate_raw_fish`, `shot_pistol_first_time`, `slept_in_house` — сделал ли действие впервые
- `saw_flashback_war`, `saw_ending_omen` — увидел ли ключевую сцену

**Числовые счётчики** (сколько раз):
- `times_visited_pyaterochka`
- `times_talked_to_cashier`
- `total_shots_fired`
- `days_survived`

**Строковые статусы** (перечисления):
- `time_of_day` : "dawn" | "day" | "dusk" | "night"
- `weather` : "clear" | "fog" | "rain"
- `alexei_last_meal` : "none" | "sandwich" | "vodka" | ...

### 19.2 Кто читает и пишет

**Читают:**
- Диалоги в Ink (через external function `knows("flag_name")`)
- NPC-триггеры (для выбора активного диалога, но обычно логика внутри Ink)
- Пассивные реакции NPC
- Игровые механики (например, освещение зависит от `time_of_day`)

**Пишут:**
- Диалоги в Ink (через external function `set_flag("flag_name", true)` или через теги)
- Игровые действия (`shot_pistol_first_time = true` при первом выстреле)
- Игровое время (`time_of_day` меняется с течением игрового дня)

### 19.3 Правило: минимум флагов

Не создавать флаг «на всякий случай». Каждый флаг должен **реально влиять** на что-то в игре:
- ❌ `looked_at_sky` — никому не нужно
- ❌ `visited_cashier_this_hour` — избыточно
- ✅ `knows_cashier_is_monster` — меняет весь тон её диалогов
- ✅ `killed_civilian` — меняет реакции всех NPC

**Тест:** «Изменится ли что-то в мире / диалогах / реакциях если этот флаг сработает?». Если нет → флаг не нужен.

### 19.4 Именование

- **snake_case** (соответствует Ink-стилю переменных)
- **Семантичные глаголы в прошедшем времени** для булевых: `met_X`, `knows_X`, `killed_X`, `ate_X`, `saw_X`
- **times_X** для счётчиков: `times_visited_X`, `times_shot`
- Никаких сокращений: `met_cashier`, не `meet_c`

### 19.5 Реестр флагов

Все флаги — в одном месте (это критично). Два уровня:

**Файл-справочник `Assets/_Project/Docs/GameFlags.md`:**
- Таблица всех флагов с описанием и связями
- Обновляется одновременно с добавлением нового флага
- Прежде чем создать новый флаг — искать в справочнике похожий

**Код-константы `Assets/_Project/Scripts/GameFlags/GameFlagKeys.cs`:**
- Статический класс со строковыми константами
- Использовать `GameFlagKeys.MetCashier`, а не строку `"met_cashier"`
- IDE подсказывает, опечатки невозможны

### 19.6 Технические компоненты

Планируется как отдельная подсистема (реализация в отдельной сессии):

- **`GameFlags`** — обычный `[Serializable]` C# класс с методами `GetBool`, `SetBool`, `GetInt`, `SetInt`, `AddInt`, `GetString`, `SetString`, `ResetAll`. Внутри — три Dictionary по типам.
- **`GameFlagsManager`** — MonoBehaviour singleton по образцу `PersonalityManager`. Держит `GameFlags` в памяти, сохраняет в JSON (`Application.persistentDataPath/game_flags.json`), при старте загружает.
- **Ink External Functions** — регистрация `knows()`, `set_flag()`, `int_get()`, `int_add()` при `StartDialogue`. Ink вызывает эти функции напрямую, они читают/пишут `GameFlagsManager.Instance.Flags`.

### 19.7 Порядок реализации

**Не сейчас.** Порядок такой:
1. UI диалога (текущая работа)
2. NPC-триггер (следующая)
3. Мост Ink ↔ PersonalityManager (первый пример работы Ink с C#)
4. **Полная система GameFlags** (после того как первый мост работает)
5. Использование в реальных диалогах

## 20. Атмосферные проявления путей

**Философия:** одно и то же действие в мире даёт **разный опыт** в зависимости от Житейской шкалы игрока. Это фирменный приём для передачи внутреннего состояния героя через мир, а не через явные шкалы на экране.

**Не сюжет, а атмосфера.** Не главные механики, а маленькие камешки, задающие тон.

**Примеры (стартовый список, пополняется):**

### 20.1 Сэндвичи с сырой рыбой [пример-эталон]

**Что:** на прилавке Пятёрочки лежит несколько сэндвичей — тупо сырая рыба на хлебе. Норма для деревни, все едят.

**Реакция героя зависит от Житейской шкалы:**
- В начале пути / умеренное состояние → игрок пробует → героя рвёт. Явное «это несъедобно».
- При Фатализме > +60 (Смирение) → игрок пробует → сэндвич кажется нормальным блюдом. Внутренний монолог типа «а неплохо, солоновато только». Никакой рвоты.

**Что игрок узнаёт:** он уже не тот человек, что приехал в глушь. Мир его принял, он принял мир.

**Пример диалога/действия при пробе:**

Житейская < +60:
```
Ты откусываешь.
Холодная рыба липнет к нёбу. Рыбий жир течёт по подбородку.
Тебя рвёт прямо у прилавка.
Кассирша молча протягивает салфетку.
```

Житейская > +60:
```
Ты откусываешь.
Солоновато. Свежо.
Мысль: не понимаю, чего я боялся.
Пожимаешь плечами, доедаешь до конца.
Кассирша кивает.
```

### 20.2 Идеи для расширения (не реализовывать, копилка)

- **Чайки** — в начале раздражают криком, при высоком Фатализме их крики звучат как чужой разговор, потом можно понимать смысл.
- **Туман** — в начале ощущается как плохая погода, при Смирении — как покрывало, уютно.
- **Молчание кассирши** — в начале пугает, потом становится тёплым.
- **Своё отражение** — по мере пути меняется, что видишь: усталость / пустоту / что-то незнакомое / хищный интерес.
- **Запах моря** — от «противный» до «родной».

### 20.3 Правила проектирования атмосферных проявлений

- **Опциональны** — не блокируют прогресс. Игрок может не заметить.
- **Награда за исследование и путь** — те кто увидели, поняли что-то важное о герое.
- **Немного** — не должно быть на каждом углу, иначе теряется вес.
- **Через тело и восприятие**, а не через явный текст «ты стал фаталистом» — показ, не сказ.
- Как минимум **2-3 ключевых** в вертикальном срезе, для чувства концепции. Расширять постепенно.

## 21. Ink-теги и обработчики (Tag Dispatcher)

**Философия:** Ink поддерживает **теги** на репликах и выборах — короткие метки для передачи команд в игру. Сценарист (разработчик) пишет теги прямо в `.ink`, а игра автоматически выполняет соответствующие действия.

Это **главный инструмент связи диалогов с миром** — анимации, звуки, визуальные эффекты, изменения состояния.

### 22.1 Как это выглядит для сценариста

```ink
=== angry_moment ===
Кассирша поднимает голову. #anim:cashier_angry #sound:cashier_gasp
"Хватит уже!" #text_color:red #shake:medium

* [Отступить] #anim:player_flinch
    -> retreat

* [Не двигаться] #anim:player_calm
    -> stay
```

Разработчик пишет **теги как естественную часть сценария**, не думая о коде. Каждый тег — команда «сделать что-то в мире».

### 22.2 Стандартный набор тегов (стартовый список, расширяется)

**Персонажи и анимации:**
- `#speaker:npc_id` — кто говорит (для подсветки, для истории)
- `#anim:animation_name` — запустить анимацию на текущем NPC
- `#player_anim:action` — запустить анимацию на игроке
- `#portrait:emotion` — портрет NPC с указанной эмоцией (если будут портреты)

**Аудио:**
- `#sound:sfx_name` — воспроизвести звуковой эффект
- `#music:track_name` — сменить фоновую музыку
- `#music_stop` — остановить музыку
- `#voice:line_id` — озвучка реплики (если будет)

**Визуал текста:**
- `#text_color:red|muted|whisper|...` — окраска текста реплики
- `#font_size:big|small` — размер шрифта
- `#typing_speed:slow|fast` — скорость печати (если делаем typewriter)

**Эффекты экрана:**
- `#shake:small|medium|big` — тряска камеры
- `#flash:color` — вспышка экрана
- `#pause:seconds` — пауза перед следующей репликой

**Игровое состояние:**
- `#shift:AXIS:±N` — изменить шкалу Личности (см. §5)
- `#set_flag:flag_name` — установить флаг GameFlags (см. §19)
- `#inc:counter_name` — увеличить счётчик GameFlags
- `#event:custom_thing` — произвольный игровой ивент (для нестандартных случаев)

**Список расширяется по мере разработки.** Новый тип эффекта → новый Handler → работает во всех диалогах.

### 22.3 Архитектура (Open/Closed Principle)

**Ключевой принцип:** добавление нового типа тега **не должно требовать изменения существующего кода**. Только добавление нового файла.

```
DialogueTagDispatcher (диспетчер)
    ├── AnimationTagHandler    → "anim:*"
    ├── SoundTagHandler        → "sound:*"
    ├── TextColorTagHandler    → "text_color:*"
    ├── CameraShakeTagHandler  → "shake:*"
    ├── ShiftTagHandler        → "shift:*"
    ├── FlagTagHandler         → "set_flag:*"
    └── ... (добавляем новые Handler-ы)
```

**Каждый Handler:**
- Знает **свой префикс тега** (например `anim:`)
- Умеет **распарсить остаток** (например `cashier_angry`)
- Умеет **выполнить действие** (найти Animator, запустить триггер)

**Диспетчер:**
- Хранит список всех Handler-ов
- При событии `InkDialogueRunner.OnDialogueContinued` читает текущие теги
- Прогоняет каждый тег через всех Handler-ов
- Тот, кто может обработать — выполняет

### 22.4 Файлы (спецификация на будущее)

Планируется создать в отдельной сессии:

- `IDialogueTagHandler.cs` — интерфейс: `bool CanHandle(string tag)`, `void Execute(string tag, DialogueContext context)`
- `DialogueTagDispatcher.cs` — MonoBehaviour singleton, держит список хендлеров, подписан на Runner
- Конкретные Handler-ы — по одному файлу на тип:
  - `AnimationTagHandler.cs`
  - `SoundTagHandler.cs`
  - `TextColorTagHandler.cs`
  - и т.д.

**Итого:** ~5 файлов базовой инфраструктуры + новый файл на каждый новый тип тега (30-50 строк).

### 22.5 Порядок реализации

Согласно дорожной карте (§18):

1. Сначала сделать мост Ink ↔ Personality
2. Первый Handler = `ShiftTagHandler` (для `#shift:ZHIT:-3`) — как proof of concept
3. Второй = `FlagTagHandler` (для `#set_flag:...`) — интеграция с GameFlags
4. **Потом** — полноценная система Handler-ов для визуала и звука

Не пытаться делать всё сразу. Каждый Handler — отдельная маленькая работа.

### 22.6 Правило для сценариста

**Прежде чем использовать новый тег в .ink** — убедиться что для него есть Handler. Иначе тег просто проигнорируется (не ошибка, но и не сработает).

**Список поддерживаемых тегов** будет вестись в `Assets/_Project/Docs/InkTags.md` по аналогии с `GameFlags.md` (см. §19.5).
