# Спецификация Системы Личности v2.0

> Технический документ. Обновлён под правило KISS+YAGNI из GDD v0.4 §14.
> Основная идея: **минимум кода, максимум использования Ink** для нарративной логики.

---

## 1. Что это и зачем

Три шкалы, описывающие характер Алексея. Значения меняются от действий игрока (диалоги, поступки), влияют на доступные варианты ответа и в итоге — на финал.

**Ключевой принцип v2.0:** система хранит **только текущие значения**. Всю логику про «какой сдвиг за какое действие», «какой вариант доступен при каком значении» — держим **в Ink-скриптах** и в местах вызова, а **не в коде системы Личности**.

---

## 2. Три шкалы

Все в диапазоне **[−100, +100]**. Стартовое значение — **0**.

| Enum ID | Русское название | Минус (−100) | Плюс (+100) | Роль |
|---|---|---|---|---|
| `Zhiteyskaya` | Житейская | Эхо войны | Фатализм | Магистральная — путь, финал, сложность механических проверок |
| `Povedencheskaya` | Поведенческая | Приказ | Своеволие | Модификатор доступа + тона + **эффективный модификатор для механических проверок** (см. §5) |
| `Poetichnaya` | Поэтичная | Трезвость | Иллюзия | Косметика + гейт «глубоких» веток + визуал механик |

## 3. Архитектура (минимальная)

Три файла в `Assets/_Project/Scripts/Personality/`:

### 3.1 `PersonalityAxis.cs` — enum

```
public enum PersonalityAxis
{
    Zhiteyskaya,
    Povedencheskaya,
    Poetichnaya
}
```

С XML-комментариями на русском.

### 3.2 `PersonalityData.cs` — данные

Обычный C# класс с атрибутом `[System.Serializable]` (не MonoBehaviour, не ScriptableObject).

**Поля:**
- 3 приватных int с `[SerializeField]`, все стартуют в 0

**Методы:**
- `int Get(PersonalityAxis axis)` — вернуть значение
- `void Set(PersonalityAxis axis, int value)` — установить (клэмп внутри)
- `void Add(PersonalityAxis axis, int delta)` — прибавить (клэмп внутри)
- `void ResetAll()` — обнулить все три

**Клэмп через `Mathf.Clamp(value, -100, 100)` — хардкод, не через константы.**

Реализация Get/Set через switch по enum.

### 3.3 `PersonalityManager.cs` — жизненный цикл + сохранение

MonoBehaviour с singleton-паттерном (`Instance`). Единственный компонент, который вешается в сцене.

**Поля:**
- `[SerializeField] string _saveFileName = "personality.json"`
- `public PersonalityData Data { get; private set; }`

**Жизненный цикл:**
- `Awake()`: устанавливает Instance, создаёт свежий `PersonalityData` (все шкалы = 0). Если файл сохранения существует — загружает из него.

**Публичные методы:**
- `Save()` — сериализовать Data в JSON, записать в `Application.persistentDataPath/personality.json`
- `Load()` — прочитать файл, десериализовать в Data. Если файла нет — оставить как есть.
- `NewGame()` — Data.ResetAll(), удалить файл сохранения если существует.

## 4. Модель сохранений

**Ключевое правило (утверждено в v0.4):**

> При старте игры все шкалы = 0. Если есть файл сохранения — он **подставляет** накопленные значения. Если файла нет — играется свежая игра.

**Никаких ScriptableObject-ов с runtime-данными.** Data живёт только в памяти. Файл на диске — единственный носитель между сессиями.

**Место сохранения:** `Application.persistentDataPath` — Unity сама определяет правильную папку для платформы.

**Формат:** JSON через `JsonUtility` — встроен в Unity, простой, читаемый глазами.

## 5. Правила фильтрации диалоговых выборов

**Живут не в C#, а в Ink-скриптах** через теги на репликах. C#-система только предоставляет **API чтения значений** (`PersonalityManager.Instance.Data.Get(...)`).

### Формат Ink-тегов (планируется, реализация в сессии интеграции Ink)

На выборах в Ink-скрипте будут теги:

```ink
* [Молчать] #tone:neutral #shift:ZHIT:+1
* [Ну быстрее!] #tone:echo #shift:ZHIT:-3
* [Спешить некуда...] #tone:fatalism #shift:ZHIT:+3
```

**Ink-мост в C#** (будущая сессия):
- Читает теги при показе выборов
- Проверяет условия доступности (через API `PersonalityManager`)
- Скрывает недоступные варианты
- При выборе — применяет сдвиг через `PersonalityManager.Instance.Data.Add(...)`

### Логика доступности (реализуется в Ink-мосте)

Пусть `Z = |Data.Get(Zhiteyskaya)|`, `sign = знак Zhiteyskaya`, `pole` — полярность варианта из тега.

- `Z ≤ 20` → все варианты доступны
- `20 < Z ≤ 60` → нейтральный + вариант «в тон» (sign совпадает с pole)
- `Z > 60` → только вариант «в тон»

**Модификатор от Поведенческой:**
- Если `20 < Z ≤ 60` (противоположный вариант скрыт) **И** `|Povedencheskaya| > 40` **И** знаки Житейской и Поведенческой противоположны → **разблокировать противоречивый вариант**

**«Глубокие» диалоги** (тег `#deep_dialogue` на узле):
- Все правила применяются к **Поэтичной**, а не Житейской
- Модификатор от Поведенческой не работает (Поведенческая не про мышление)

**Тон реплик** (окраска, не влияет на смысл):
- От Поведенческой при `|Pov| > 40`: сухой (Приказ) / грубый (Своеволие)
- От Поэтичной при `|Poet| > 40`: приземлённый (Трезвость) / поэтичный (Иллюзия)
- Реализация — через альтернативные тексты в Ink с проверкой значений

## 5.5 Формула эффективной шкалы (для механических проверок)

**Для механических проверок** (мини-игра стрельбы, будущие проверки реакции и т.п.) применяется **не реальная Житейская, а эффективная**:

```
эффективная_Житейская = реальная_Житейская + (Поведенческая × 0.2)
```

Затем клэмп к [−100, +100].

**API-метод в `PersonalityData` (добавить в будущей сессии, когда появится стрельба):**
```
public int GetEffective(PersonalityAxis axis)
{
    if (axis == PersonalityAxis.Zhiteyskaya)
    {
        int povBonus = _povedencheskaya / 5; // ×0.2 через целочисленное деление
        return Mathf.Clamp(_zhiteyskaya + povBonus, -100, 100);
    }
    return Get(axis); // для остальных — просто реальное значение
}
```

**Пока не реализуется** — добавим при работе над мини-игрой стрельбы.

## 6. Величина сдвигов — эмпирически, не в коде

**Никаких `SmallShiftAmount`, `MediumShiftAmount`, `LargeShiftAmount` констант.**

Величина сдвига определяется в **месте применения**:
- В Ink-теге: `#shift:ZHIT:+3` → сдвиг +3
- В C#-действии (например, убийство): `PersonalityManager.Instance.Data.Add(Zhiteyskaya, -20)` → значение прямо в вызове

Так проще балансить (не надо помнить, какая константа что означает), проще читать (в диалоге сразу виден вес выбора).

**Рекомендуемые диапазоны для баланса** (из §9 GDD, для справки, не константы):
| Тип действия | Сдвиг |
|---|---|
| Малый выбор в тон | ±1..±3 |
| Значимый моральный | ±5..±8 |
| Ключевое сюжетное | ±10..±15 |
| Убийство мирного | −20 |
| Убийство дружелюбного монстра | −15 |
| Пощада агрессивного | +3 |
| Ритуал (курение, наблюдение) | +1 |

## 7. Тестирование системы (сессия 5, первая реализация)

**Модифицированный `TestInteractable`:**
- Поле `[SerializeField] bool _applyPersonalityShift = false` (галочка в инспекторе)
- Поле `[SerializeField] int _shiftAmount = 1`
- При `Interact()`, если галочка включена — прибавить `_shiftAmount` ко всем трём шкалам через `PersonalityManager.Instance.Data.Add(...)` и залогировать состояние

**Проверка работоспособности:**
1. Play → нажимать E на кубике многократно
2. Значения должны расти в лог
3. После достижения +100 → должны застревать (проверка клэмпа)
4. Save (через удобный триггер — обсудим на реализации) → закрыть Play → снова Play → значения должны подтянуться
5. NewGame → значения обнуляются, файл удаляется
